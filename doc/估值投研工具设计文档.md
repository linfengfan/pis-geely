# 估值投研工具 — 设计文档

**日期**：2026-05-10
**状态**：初稿，待评审
**技术栈**：TypeScript 全栈（Next.js 前端 + Next.js API 后端）
**数据库**：PostgreSQL + Prisma ORM（会话存储、完成估值记录）

---

## 1. 项目概述与目标

### 1.1 核心目标

构建一个多轮对话式估值投研工具，用户输入公司名称后，系统通过多 Agent 协作完成企业内在价值测算，最终输出结构化投研分析报告。

### 1.2 一期范围

| 范围 | 说明 |
|------|------|
| 数据来源 | 用户手动填写，一期不接外部财务数据库 |
| 宏观数据 | 用户手动填写（CN10Y、USD/CNY、社融增速等） |
| 时间维度 | 仅支持当前年度财报估值，无历史数据 |
| 投研报告 | 有机叙事文本 + 支撑图表，无结论判断、无执行策略、无股价走势 |
| 附件解析 | 预留扩展点（MD 年报解析），二期实现 |
| 公司搜索 | WebSearch 优先，本地知识库兜底（扩展点） |
| 会话管理 | 进行中会话不持久化；已完成估值存库，展示为历史卡片 |

## 2. 系统架构

### 2.1 Multi-Agent 架构

```
┌──────────────────────────────────────────────────────┐
│            ValuationOrchestrator（Host Agent）          │
│                   主控编排层                             │
│  - 维护会话状态                                         │
│  - 路由决策（调用哪个子 Agent）                           │
│  - 参数完整性校验                                        │
│  - 响应聚合，统一返回给前端                               │
└────────────────┬───────────────────────────────────┘
                 │ Tool Call
     ┌───────────┼───────────┬────────────┐
     ▼           ▼           ▼            ▼
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐
│Valuation │ │Valuation │ │Document  │ │   WebSearch  │
│Research  │ │Analysis  │ │Parser    │ │   (工具)     │
│Agent     │ │Agent     │ │Agent     │ │              │
│子 Agent   │ │子 Agent   │ │子 Agent   │ │ 搜索公司信息 │
└──────────┘ └──────────┘ └──────────┘ └──────────────┘
```

### 2.2 各 Agent 职责

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **ValuationResearchAgent** | 根据公司名称研判结构，选模型，定义参数 Schema | 公司名称 | `{ model, schema, reasoning }` |
| **ValuationAnalysisAgent** | 参数齐备后执行估值计算，生成投研分析报告 | 公司名称 + Schema + 用户填入的参数 + 宏观数据 | `{ analysisReport }` |
| **DocumentParserAgent** | 解析用户上传的 MD 年报，提取结构化指标 | MD 文件内容 | `{ extractedFields, confidence, warnings }` |

### 2.3 技术实现

- 所有 Agent 基于 **Anthropic Claude API**（Messages API）
- Orchestrator 与子 Agent 之间通过 **Tool/Function Call** 机制通信
- 每个子 Agent 拥有独立 System Prompt，职责边界清晰
- 前端不调用 LLM，所有推理在后端完成

## 3. 前端页面布局

### 3.1 整体布局

```
┌─────────────────────────────────────────────────────────┐
│                    Header                              │
├─────────────────────────────┬─────────────────────────┤
│                             │                         │
│     动态参数表单区            │     Agent 聊天区          │
│     （左侧，宽度 40%）         │     （右侧，宽度 60%）     │
│                             │                         │
│  根据 Schema 动态渲染         │  对话式交互              │
│  用户填写参数                 │  展示模型反馈            │
│                             │  "使用真实数据" 按钮     │
│                             │                         │
└─────────────────────────────┴─────────────────────────┘
```

### 3.2 左侧：动态参数表单区

- 根据 **ValuationResearchAgent 返回的 Schema** 动态渲染表单字段
- Schema 由后端定义后一路透传给前端，前端不存规则、不推导字段
- 表单字段包含：公司信息、财务数据、宏观数据、估值参数
- 当前股价字段用于计算安全边际
- 缺失参数以高亮（蓝色边框）展示

### 3.3 右侧：Agent 聊天区

- 展示用户与模型之间的多轮对话记录
- 包含"**使用真实数据**"按钮：将当前左侧表单数据注入对话上下文，发给 Orchestrator
- 用户可通过对话引导模型行为（如解释参数含义、询问分析逻辑）
- 对话区域展示结构化反馈（如缺失参数提示）

### 3.4 两个数据发送渠道

| 渠道 | 触发方式 | 行为 |
|------|----------|------|
| 表单提交 | 用户点左侧表单提交按钮 | 将表单数据发给 Orchestrator |
| 使用真实数据 | 用户点右侧"使用真实数据"按钮 | 将当前表单数据注入对话上下文，发给 Orchestrator |

两条渠道最终都指向同一个处理流程：Orchestrator 校验参数 → 调用对应子 Agent。

### 3.5 已完成估值卡片

- 估值完成的 Session（带 sessionId）存入数据库
- 前端展示为"已估值公司卡片"列表，供用户回顾
- 不做历史消息留存；用户关闭页面再回来，重新开始

## 4. 核心交互流程

```
用户输入公司名称
       ↓
Orchestrator 接收消息
       ↓
ValuationResearchAgent（研判 + 选模型 + 定义 Schema）
       ↓
返回 { type: 'schema_defined', content: { model, schema, reasoning } }
       ↓
前端：按 Schema 渲染动态表单
       ↓
用户填写参数 → 提交（渠道 A 或渠道 B）
       ↓
Orchestrator 接收参数
       ↓
[参数校验] Schema vs 已填值
       ├── 缺失 → 返回 { type: 'missing_params', content: { missing: [...] } }
       │         前端高亮缺失字段，用户继续填写
       └── 完整 → 调用 ValuationAnalysisAgent
                    ↓
            返回 { type: 'analysis_report', content: { ... } }
                    ↓
            前端渲染为一张结果卡片
```

**用户上传 MD 年报时的流程：**

```
用户上传 MD 文件
       ↓
DocumentParserAgent 解析
       ↓
返回 { extractedFields, confidence, warnings }
       ↓
前端：自动将提取的字段填入表单（待确认态），高亮显示
       ↓
用户核对、修改后提交
```

## 5. 服务端 → 前端：响应类型定义

Orchestrator 返回给前端的统一响应结构为 `{ type, content }`，前端根据 `type` 决定渲染方式。

### 5.1 响应类型总览

| type | 含义 | 前端行为 |
|------|------|----------|
| `chat` | 普通文本消息 | 追加到右侧聊天区 |
| `schema_defined` | 模型和参数 Schema 已定义 | 按 Schema 重新渲染左侧表单 |
| `missing_params` | 缺失参数列表 | 高亮左侧表单中的缺失字段 |
| `analysis_report` | 完整估值结果 | 渲染为一张结果卡片 |
| `error` | 异常错误 | 展示错误提示，不继续链路 |

### 5.2 `type: 'chat'`

```typescript
{
  type: 'chat',
  content: {
    text: string        // 模型输出的普通文本
  }
}
```

### 5.3 `type: 'schema_defined'`

```typescript
{
  type: 'schema_defined',
  content: {
    model: string,           // 选定的估值模型，如 'DCF'、'PE'
    schema: ParameterSchema, // 参数 Schema（见 5.6 节）
    reasoning: string       // 模型选择理由
  }
}
```

### 5.4 `type: 'missing_params'`

```typescript
{
  type: 'missing_params',
  content: {
    missing: {
      field: string,         // 字段名（与 Schema 中的 name 对应）
      label: string,         // 中文标签
      reason: string         // 缺失原因描述（可选）
    }[]
  }
}
```

### 5.5 `type: 'analysis_report'`

```typescript
{
  type: 'analysis_report',
  content: {
    // 卡片顶部：核心计算结果
    model: string,               // 估值模型名称
    intrinsicValue: number,      // 内在价值（元/股）
    marginOfSafety: number,      // 安全边际（%）

    // 卡片中部：完整投研分析文本
    // Agent 生成的有机叙事，包含公式、数据切片、诊断过程、结论描述
    // 使用 Markdown 格式，可包含数学公式、代码块、表格等
    analysisText: string,

    // 卡片底部支撑数据：用于前端渲染图表
    metrics: {
      name: string,              // 指标名称
      value: number,             // 公司值
      unit: string,              // 单位
      industryAvg: number        // 行业均值（用于对比图表）
    }[]
  }
}
```

**前端渲染方式：**

- 整体渲染为 **一张结果卡片**
- 顶部数字区：model / intrinsicValue / marginOfSafety
- 中部：analysisText 以格式化文本渲染（支持 Markdown）
- 底部：metrics 数组驱动柱状图（公司 vs 行业均值对比）

### 5.6 `ParameterSchema` 参数 Schema 定义

Schema 是链路各方共同遵守的"合同"，由 ValuationResearchAgent 定义，透传全链路。

```typescript
interface ParameterSchema {
  model: string
  description: string                    // 模型说明
  requiredParams: ParameterDefinition[]  // 必需参数
  optionalParams: ParameterDefinition[] // 可选参数
}

interface ParameterDefinition {
  name: string        // 字段名（英文，代码中使用）
  label: string       // 中文标签
  type: 'number' | 'string'
  unit?: string       // 单位（如 '亿元'、'%'）
  description: string // 参数说明（给用户看）
}
```

**示例：DCF 模型 Schema**

```typescript
{
  model: 'DCF',
  description: '现金流折现模型，适用于经营现金流充沛的成熟企业',
  requiredParams: [
    { name: 'freeCashFlow', label: '自由现金流（亿元）', type: 'number', unit: '亿元' },
    { name: 'wacc', label: '折现率 WACC（%）', type: 'number', unit: '%' },
    { name: 'terminalGrowthRate', label: '永续增长率（%）', type: 'number', unit: '%' },
    { name: 'projectionYears', label: '预测年数', type: 'number' }
  ],
  optionalParams: [
    { name: 'netDebt', label: '净负债（亿元）', type: 'number', unit: '亿元' }
  ]
}
```

### 5.7 `type: 'error'`

```typescript
{
  type: 'error',
  content: {
    code: string,     // 错误码
    message: string   // 展示给用户的错误信息
  }
}
```

错误在前端展示提示，**不继续 agent 链路**。

## 6. 错误处理

| 场景 | 处理方式 |
|------|----------|
| 公司名称搜不到 | 返回 `error` type，前端展示提示 |
| LLM 返回格式异常 | 返回 `error` type，前端展示提示 |
| 网络超时 | 返回 `error` type，前端展示提示 |
| 参数缺失 | 返回 `missing_params` type，引导用户补充 |
| Agent 调用失败 | 返回 `error` type，前端展示提示 |

所有错误均在 Orchestrator 层统一处理，前端拦截，不往下串链路。

## 7. 会话管理

| 状态 | 处理方式 |
|------|----------|
| 进行中会话 | 数据存内存，不持久化；用户关闭页面，数据清除 |
| 已完成估值 | sessionId + 结果写入数据库，前端展示为历史卡片 |
| 用户再次访问 | 关闭页面重新开始（不恢复上下文）；历史卡片可点击重新发起 |

## 8. 扩展点规划

### 8.1 附件解析（B 期）

- 用户上传 MD 格式年报
- 调用 DocumentParserAgent，Token 消耗控制在 1,500~15,000 tokens（仅解析财务三表）
- 解析结果自动填入表单（待确认态），高亮显示
- 不占用 Agent 聊天主流程（独立路径）

### 8.2 本地知识库兜底

- WebSearch 优先
- 若搜索失败或结果不充分，切换本地知识库检索
- 接口统一，具体实现作为扩展点预留

### 8.3 新增估值模型

- 在 Schema 配置中新增一条模型定义
- 不需要修改前端代码
- 不需要修改 Orchestrator 路由逻辑
- ValuationResearchAgent Prompt 中预置所有可用模型定义

### 8.4 历史趋势（二期）

- 当用户累计提交多年数据后，可绘制内在价值随年度变化的折线图
- 数据源：用户多年提交 或 外部财务数据库（Tushare / Wind）

## 9. 投研分析报告示例

以下为中国石油案例的投研分析文本风格示例（由 ValuationAnalysisAgent 生成）：

---

**手术台的无影灯已将这 8 个关键的周期参数彻底打亮。**

在 85 美元的布伦特原油高气压下，中国石油的躯体展现出了强烈的"抗周期变异"特征。我们将这 8 个参数注入资本离心机，对其进行绝对冷酷的物理切割：

**一、吞金兽的"排异测试"：彻底蜕变的造血器官**

- 血液泵出量（经营现金流）：4125.10 亿元
- 血液回流消耗（资本性支出）：2690.89 亿元
- 真实造血能力（自由现金流 FCF）：1434.21 亿元

**定性诊断：** 这具重资产躯体已每年净输送超 1400 亿元纯血，从"吞金兽"变异为一台"巨型周期印钞机"。

**二、核心骨架切割：PB-ROE 绝对估值**

- 可持续增长率 g = ROE × (1 - 派息比率) = 8% × 45.3% = **3.624%**
- 目标 PB = (ROE × 派息比率) / (r - g) = **0.686 倍**
- 单股内在价值 V = 0.686 × 8.666 (BPS) = **5.94 元/股**

**三、绝对防御底线：DDM 现金分红装甲**

- 每股绝对分红 DPS = 0.693 × 54.7% = **0.379 元**
- 零增长极寒防线：0.379 / 10% = **3.79 元**
- 抗通胀防线：0.379 / 8% = **4.74 元**

**手术结论：**

在 85 美元油价和 10% 目标回报率的双重重力下，中国石油的真实内在价值中枢被锁定在 **5.94 元**。当前市场给予的估值已远超中枢，仅当宏观地缘危机解除、原油价格预期崩塌、股价被重力砸回 **4.15 元** 深渊时，才具备建仓物理赔率。

---

**设计说明：**

- analysisText 字段承载以上格式的文本（Markdown + 数学公式）
- metrics 数组提供图表支撑数据（如 FCF、ROE、派息比率、DPS 等）
- Agent 的 System Prompt 需明确要求按此叙事风格生成报告

## 10. 数据库设计（Prisma）

### 10.1 新增 Session 表

```prisma
model ValuationSession {
  id          String   @id @default(cuid())
  companyName String
  model       String?
  status      SessionStatus @default(IN_PROGRESS)
  schema      Json?              // ValuationResearchAgent 返回的 Schema
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  analysis    ValuationAnalysis?
  turns       ConversationTurn[]
}

enum SessionStatus {
  IN_PROGRESS
  COMPLETED
}

model ConversationTurn {
  id          String   @id @default(cuid())
  sessionId   String
  role        String               // 'user' | 'assistant'
  type        String               // 响应 type 字段
  content     Json                 // 响应 content 字段
  createdAt   DateTime @default(now())

  session     ValuationSession @relation(fields: [sessionId], references: [id], onDelete: Cascade)
}

model ValuationAnalysis {
  id              String   @id @default(cuid())
  sessionId       String   @unique
  model           String
  intrinsicValue  Float
  marginOfSafety  Float
  analysisText    String   @db.Text
  metrics         Json     // metrics 数组
  createdAt       DateTime @default(now())

  session         ValuationSession @relation(fields: [sessionId], references: [id], onDelete: Cascade)
}
```

### 10.2 现有表保留

- `Company` — 公司基础信息（未来扩展）
- `Portfolio` / `Holding` — 组合管理（保持不变）
- `MacroSnapshot` — 宏观数据快照（保持不变）

## 12. 技术约束

- 前端不调用 LLM，所有推理在后端完成
- 后端不存 Agent Prompt 内容作为静态配置，统一管理
- Schema 名 = 代码字段名 = LLM 理解的变量名，三者必须完全一致
- Agent 使用 Anthropic Claude Sonnet 4（成本与能力平衡）
- 一期不引入 Python 服务，全栈 TypeScript
